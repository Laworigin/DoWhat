## 支柱二：模块即边界

每个模块有清晰的职责、接口和生命周期，模块之间通过显式接口通信，禁止隐式耦合。

### DO

- **遵循单一职责原则**：一个文件只做一件事。数据库操作在 `database.ts`，截屏逻辑在 `capturer.ts`，UI 组件在 `components/`
- **使用注册表模式管理同类模块**：所有 IPC handler 集中注册，所有定时任务集中管理，便于全局审视
- **新增功能模块时遵循标准目录结构**：
  ```
  src/main/       → 主进程逻辑（数据库、截屏、系统交互）
  src/renderer/   → 渲染进程（UI 组件、hooks、i18n）
  src/preload/    → 预加载脚本（IPC 桥接）
  ```
- **重型依赖使用动态 import 延迟加载**，避免拖慢启动速度
- **模块导出必须显式**：使用命名导出，禁止 `export default` 导出匿名对象
- **循环依赖必须通过提取共享类型到独立文件来打破**，禁止用 `require()` 绕过

### DON'T

- 禁止在渲染进程中直接引用 Node.js 模块（`fs`、`path`、`child_process`）
- 禁止在一个文件中混合主进程逻辑和 UI 逻辑
- 禁止在组件内部直接调用 `ipcRenderer.invoke`，必须通过 `window.api` 桥接层
- 禁止在启动路径上同步加载非必要模块
- 禁止跨模块直接访问另一个模块的内部状态，必须通过导出的函数接口
