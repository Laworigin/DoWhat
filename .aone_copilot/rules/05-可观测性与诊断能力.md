## 支柱五：日志即基建 + 打包即交付

关键操作必须可追踪、可诊断。构建产物必须干净、可复现。

### 日志规范

#### DO

- **使用带模块前缀的结构化日志**，便于过滤和定位
  ```typescript
  console.log('[DB] Database initialized at:', dbPath)
  console.log('[Capturer] Screenshot saved:', filePath)
  console.error('[Migration] Failed to update paths:', error.message)
  ```
- **关键生命周期事件必须记录日志**：应用启动、数据库初始化、定时任务启动/停止、IPC 调用失败
- **日志必须包含足够的上下文信息**：操作名称、关键参数值、耗时（对性能敏感操作）
- **敏感信息必须脱敏**：API Key 只显示前 4 位，用户路径用 `~` 替代 home 目录

#### DON'T

- 禁止在生产代码中使用无前缀的 `console.log('test')` 或 `console.log(variable)`
- 禁止在日志中输出完整的 API Key、密码、token
- 禁止在高频循环中输出日志（如每次截屏都打印完整路径），应使用采样或汇总

### 构建与交付规范

#### DO

- **每次打包前验证产物干净性**：检查 asar 包中不包含 `*.db`、`snapshots/`、`.env` 等用户数据
- **签名流程必须自动化**：逐层签名脚本化，从内到外（.node → .dylib → frameworks → helpers → main app）
- **版本号必须在 `package.json` 中统一管理**，打包配置引用而非硬编码

#### DON'T

- 禁止手动修改 dist 目录下的文件后直接发布
- 禁止跳过签名验证步骤
- 禁止在打包配置中使用通配符包含整个项目目录
