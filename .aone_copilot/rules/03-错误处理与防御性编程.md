## 支柱三：错误即信号

错误是系统健康的信号，必须被分类、被处理、被追踪。每个 catch 块都必须有明确的处理策略。

### DO

- **为不同的错误场景定义自定义错误类**，携带结构化的上下文信息
  ```typescript
  class DatabaseError extends Error {
    constructor(
      message: string,
      readonly operation: string,
      readonly table: string
    ) {
      super(`[DB:${table}] ${operation} failed: ${message}`)
    }
  }
  ```
- **在模块边界处统一处理错误**：IPC handler 是主进程的错误边界，组件的 ErrorBoundary 是渲染进程的错误边界
- **区分三种错误处理策略并显式选择**：
  - **快速失败**：关键初始化错误（数据库连接失败）→ 立即退出并提示用户
  - **优雅降级**：非关键功能错误（截屏失败）→ 记录日志，跳过本次，继续运行
  - **重试恢复**：瞬态错误（网络超时）→ 带退避的重试机制
- **文件系统操作必须先检查再执行**：`fs.existsSync()` 在前，操作在后
- **所有 `catch` 块必须记录错误上下文**，至少包含：操作名称、输入参数、错误消息

### DON'T

- 禁止空 catch 块：`catch (e) {}` 是最危险的代码
- 禁止用 `console.error` 后继续执行而不做任何恢复处理
- 禁止在循环中不加 try-catch 地执行可能失败的操作，一次失败不应中断整个批处理
- 禁止抛出裸字符串：`throw 'something went wrong'`，必须抛出 Error 实例
- 禁止在 catch 中吞掉原始错误信息，必须保留 cause 链
