## 支柱一：可读性 — 让系统对 Agent/开发者透明可理解

> Agent 在每次会话开始时，对你的项目几乎一无所知。你不明确告诉它项目是什么、规则是什么、什么算完成，它就会按自己的"默认世界观"来做事。

### 核心理念

可读性不是"写注释"，而是**让整个系统对 Agent/开发者可被理解**。就像 README 是给人看的项目说明书，Harness 的可读性层是给 Agent 看的项目说明书。

### 三层信息架构（渐进披露，不是一次灌满）

```
第一层：AGENTS.md / rules 文件（始终加载）
  → 项目是什么、技术栈、关键规则、验证命令
  → 精简到 Agent 每次都能读完

第二层：docs/ 目录（按需查阅）
  → 详细设计文档、架构决策记录、API 规范
  → Agent 需要时主动查阅，不预加载

第三层：代码自文档化（阅读代码时自然获取）
  → 类型签名、命名规范、模块结构
  → 好的代码本身就是文档
```

**关键原则：全量灌输不如渐进披露。** 把所有文档一股脑塞给 Agent，效果反而更差。

### 项目说明书必须包含的三类信息

#### WHAT — 项目是什么

- 技术栈：Electron + React + TypeScript + better-sqlite3
- 项目类型：桌面应用（macOS），定时截屏 + AI 分析用户行为
- 目录结构：`src/main/`（主进程）、`src/renderer/`（渲染进程）、`src/preload/`（IPC 桥接）

#### HOW — 怎么开发和验证

- 安装依赖：`npm install`
- 开发运行：`npm run dev`
- 打包构建：`npm run build:mac`
- 类型检查：`npx tsc --noEmit`

#### RULES — 哪些规则不能碰

- 渲染进程禁止直接引用 Node.js 模块，必须通过 `window.api` 桥接
- 用户数据必须存储在 `app.getPath('userData')`，禁止用 `process.cwd()`
- 所有 IPC 通道必须有类型定义
- 数据库文件禁止打包进应用产物

### 代码自文档化标准

- **函数签名即文档**：参数类型和返回类型必须显式标注
- **命名即意图**：`migrateDataFromRootToUserData` 而非 `migrateData`
- **模块即职责**：一个文件只做一件事，文件名即职责描述
- **类型即约束**：用判别联合表达有限状态，用品牌类型防止混用
  ```typescript
  type CaptureResult =
    | { status: 'success'; imagePath: string; timestamp: number }
    | { status: 'failed'; error: string; retryable: boolean }
    | { status: 'skipped'; reason: string }
  ```

### 上下文管理（防止长任务跑偏）

- 长任务必须分阶段，每个阶段有明确的输入/输出/完成标准
- 上下文过长时，生成结构化交接文档后开启新会话继续
- 交接文档必须包含：已完成的工作、当前进度、下一步计划、注意事项
