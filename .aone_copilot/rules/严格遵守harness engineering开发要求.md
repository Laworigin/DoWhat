## Harness Engineering 开发理念总纲

> Harness Engineering 不是让模型/开发者更聪明，而是让系统真正可驾驭、可上线、可交付。

Harness 的本义是"缰绳 + 马鞍 + 挽具" — 给一匹力量很大、速度很快、但方向感不稳定的马，配上一整套能驾驭它的东西。本项目的 Harness 就是围绕开发过程构建的**约束、反馈、验证和执行控制机制**。

### 三根支柱

| 支柱 | 核心问题 | 对应规则文件 |
|---|---|---|
| **可读性** | 系统对 Agent/开发者是否透明可理解？ | `01-可读性.md` |
| **防御机制** | 关键流程是否有硬约束，而非靠自觉？ | `02-防御机制.md` |
| **反馈回路** | 错误是否被记录、被学习、形成闭环？ | `03-反馈回路.md` |

### 闭环飞轮（最核心的理念）

```
犯错/踩坑
    ↓
分析根因（是缺信息？缺约束？缺验证？缺回退？）
    ↓
更新对应层（补文档 / 补约束 / 补验证 / 补权限）
    ↓
记录到 .harness/lessons-learned.md
    ↓
下次开发自动带上 → 同类错误不再发生
    ↓
定期回顾 → 模型/工具变强后拆掉冗余约束 → 暴露新边界 → 新增新约束
```

**这个飞轮是 Harness Engineering 的灵魂。没有闭环，一切规则都只是摆设。**

### 执行原则

- 每条规则都有对应的细分文件，定义具体的执行标准
- **软约束不够，关键流程必须有硬约束** — 写在 Prompt/文档里的规则会被忘记，写在执行层的约束绕不过去
- **生成和评估必须分开** — 自己写自己评容易放水
- 每次犯错都必须更新 `.harness/lessons-learned.md`，否则同类错误会反复发生
- 每次新增功能都必须过 `.harness/verification-checklist.md` 的验证清单

### 开发前必做：Review 历史经验（硬约束）

**每次规划和开始开发新功能之前，必须先阅读 `.harness/lessons-learned.md`**，确认：

1. 本次开发涉及的模块是否有历史踩坑记录
2. 是否有相关的硬约束需要遵守（如 preload 修改需重启、process.cwd() 禁用等）
3. 验收流程是否覆盖了历史教训中暴露的盲区

**这不是可选的建议，而是硬约束。** 不 review 历史经验就开始开发，等于无视团队积累的血泪教训，同类错误必然重复发生。

```
开发前检查清单：
1. 阅读 .harness/lessons-learned.md ← 必做
2. 识别本次开发涉及的模块和历史坑点
3. 将相关约束纳入本次开发计划
4. 开发完成后，如有新踩坑，立即补充到 lessons-learned.md
```
