## 支柱二：防御机制 — 硬约束，不靠自觉

> 软约束不够，关键流程必须有硬约束。"请先做计划再执行"、"请不要跳过测试"写在 Prompt 里有用，但不可靠。写在执行层的约束，绕不过去。

### 核心理念

Harness 的防御机制不是"提醒开发者注意"，而是**在执行层拦住错误行为**。就像工厂的安全门 — 不是贴个"小心"标签，而是物理上不让你伸手进去。

### 硬约束清单（必须在执行层强制执行）

#### 约束一：改了代码必须验证

这是最低成本、最高收益的硬约束。

```
改了业务代码 → 必须跑 npx tsc --noEmit
改了 UI 组件 → 必须在开发模式下确认渲染正常
改了数据库 schema → 必须验证迁移逻辑的幂等性
改了打包配置 → 必须验证产物干净性（不含 .db、snapshots/）
```

**验证不通过，不得宣称任务完成。**

#### 约束二：环境边界不可逾越

| 边界 | 硬约束 |
|---|---|
| 数据存储路径 | 只允许 `app.getPath('userData')`，禁止 `process.cwd()` |
| 进程隔离 | 渲染进程禁止直接引用 `fs`/`path`/`child_process` |
| IPC 通信 | 必须通过 `window.api` 桥接层，禁止直接 `ipcRenderer` |
| 打包产物 | `electron-builder.yml` 必须排除 `*.db`、`snapshots/`、`.env` |
| 敏感信息 | API Key 禁止硬编码，禁止在日志中输出完整 token |

#### 约束三：防止"嘴上完成，实际没做"

每次宣称任务完成前，必须自查：

- 是否有实际的代码修改？（不能只说"我已经完成了"但没有任何 diff）
- 修改的代码是否通过了类型检查？
- 是否遗漏了导入语句？
- 是否考虑了开发环境和打包环境的差异？

#### 约束四：防止反复撞墙

同一个方案失败 **2 次**后，必须停下来换思路：

- 分析前两次失败的根因
- 提出至少一个不同的替代方案
- 如果替代方案也失败，升级为人工介入

禁止第 3 次尝试同样的方案。这和后端的熔断器是同一个思路。

### 权限边界

以下目录/文件禁止在未经确认的情况下修改：

```
.env                    → 包含 API Key 等敏感信息
electron-builder.yml    → 打包配置，改错会导致产物异常
package.json            → 依赖变更需要确认兼容性
src/main/index.ts       → 主进程入口，改动影响全局
```

### 错误处理的硬约束

- **禁止空 catch 块**：`catch (e) {}` 是最危险的代码
- **禁止抛出裸字符串**：必须抛出 Error 实例
- **禁止吞掉原始错误**：必须保留 cause 链
- **文件操作必须先检查再执行**：`fs.existsSync()` 在前，操作在后
- **数据迁移必须有幂等性保障**：重复执行不能产生副作用
