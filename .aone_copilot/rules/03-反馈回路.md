## 支柱三：反馈回路 — 让每次犯错都变成系统改进

> Harness 不是只负责"挡错"，还要负责"吃一堑长一智"。没有反馈回路的规则体系，就像没有刹车的车 — 迟早翻车。

### 核心理念

反馈回路是 Harness Engineering 的灵魂。它的目标是：**让同样的错误永远不会发生第二次**。

```
犯错/踩坑
    ↓
归因分析（是缺信息？缺约束？缺验证？缺回退？）
    ↓
更新对应层
  ├── 缺信息 → 补文档（更新 rules 文件或 docs/）
  ├── 缺约束 → 补硬约束（更新 02-防御机制.md）
  ├── 缺验证 → 补验证步骤（更新 verification-checklist.md）
  └── 缺回退 → 补回退机制（添加迁移标记、幂等检查等）
    ↓
记录到 .harness/lessons-learned.md
    ↓
下次开发自动带上 → 同类错误不再发生
```

### 自动验证回路

每次代码修改完成后，必须按顺序执行以下验证：

```
Step 1: 类型检查    → npx tsc --noEmit
Step 2: Lint 检查   → 使用 IDE 的 read_lints 检查修改过的文件
Step 3: 环境兼容性  → 确认代码在 dev 和 packaged 两种环境下都能工作
Step 4: 产物干净性  → 打包后检查 asar 不含用户数据
```

任何一步失败 → 反馈错误信息 → 修复 → 重新验证 → 直到全部通过或升级人工。

### 生成和评估分离

自己写自己评容易"放水"。关键场景下必须分离：

- **写代码**和**验证代码**是两个独立步骤，不能合并
- 修改完代码后，必须用独立的验证步骤（类型检查、lint）来确认正确性
- 对于复杂修改，建议用独立的 sub agent 做 code review

### 错误经验持久化（闭环的实体）

每次踩坑后，必须在 `.harness/lessons-learned.md` 中记录：

```markdown
## YYYY-MM-DD: 简短标题

- **问题**：发生了什么
- **根因**：为什么会发生（缺信息/缺约束/缺验证/缺回退）
- **修复**：做了什么修复
- **系统改进**：更新了哪个规则/约束/验证步骤
- **状态**：已纳入硬约束 / 已更新文档 / 待观察
```

**这不是可选的。每次犯错都必须记录，否则同类错误会反复发生。**

### 持续改进飞轮

规则体系不是一劳永逸的，它必须随着项目演化而演化：

```
定期回顾 lessons-learned.md
    ↓
识别模式：哪些错误反复出现？哪些约束已经冗余？
    ↓
更新规则体系
  ├── 反复出现的错误 → 升级为硬约束
  ├── 不再发生的错误 → 降级或移除约束
  └── 新出现的问题类型 → 新增对应的规则
    ↓
继续运行 → 继续收集经验 → 继续优化
```

### 验证清单（每次新增功能必须过）

详见 `.harness/verification-checklist.md`，包含：

- 类型安全检查
- 环境兼容性检查
- 数据隔离检查
- 错误处理检查
- 打包产物检查
