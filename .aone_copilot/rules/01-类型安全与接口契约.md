## 支柱一：类型即契约

用 TypeScript 类型系统在编译期消灭整类 bug，让错误无处藏身。

### DO

- **为所有函数签名显式标注参数类型和返回类型**，禁止依赖隐式推断的公共 API
- **使用判别联合（Discriminated Union）表达有限状态集合**，强制调用方穷举处理
  ```typescript
  type TaskResult =
    | { status: 'success'; data: TaskData }
    | { status: 'error'; error: AppError }
    | { status: 'cancelled'; reason: string }
  ```
- **为语义不同但底层类型相同的值使用品牌类型（Branded Type）**，防止混用
  ```typescript
  type TaskId = string & { readonly __brand: 'TaskId' }
  type ProjectId = string & { readonly __brand: 'ProjectId' }
  ```
- **IPC 通道的请求和响应必须有对应的类型定义**，主进程与渲染进程之间的数据交换必须类型安全
- **数据库表结构必须有对应的 TypeScript 接口**，SQL 查询结果必须经过类型断言
- **使用 Zod 等 schema 库校验外部输入**（API 响应、用户配置、文件内容），在运行时边界处验证

### DON'T

- 禁止使用 `any` 类型，必须用 `unknown` + 类型守卫替代
- 禁止用 `as` 强制类型断言绕过编译器检查，除非有充分注释说明安全性
- 禁止在 IPC handler 中直接使用 `_` 忽略事件参数的类型
- 禁止让数据库查询返回 `any`，必须定义返回类型接口
