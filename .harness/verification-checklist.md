# DoWhat 项目验证清单

> 每次新增功能或修改代码后，必须逐项检查。这是防御机制的落地文件。
> 不是所有项都需要每次检查，根据修改范围选择对应的检查项。

---

## 一、代码修改后必检（每次都要过）

- [ ] **类型检查通过**：`npx tsc --noEmit` 无错误
- [ ] **无遗漏的导入语句**：新引用的模块/类型都已 import
- [ ] **无空 catch 块**：每个 catch 都有明确的处理策略
- [ ] **无裸 console.log**：日志必须带 `[模块名]` 前缀

---

## 二、主进程修改时必检

- [ ] **路径安全**：数据存储使用 `app.getPath('userData')`，未使用 `process.cwd()`
- [ ] **环境兼容**：代码在 `is.dev` 和 `app.isPackaged` 两种环境下都能正常工作
- [ ] **IPC 类型安全**：新增的 IPC 通道有对应的类型定义
- [ ] **错误边界**：IPC handler 有 try-catch 包裹，错误信息结构化返回

---

## 三、渲染进程修改时必检

- [ ] **进程隔离**：未直接引用 `fs`/`path`/`child_process` 等 Node.js 模块
- [ ] **IPC 桥接**：通过 `window.api` 调用，未直接使用 `ipcRenderer`
- [ ] **i18n 覆盖**：新增的用户可见文案已添加到 `i18n.ts` 的 zh 和 en 翻译中
- [ ] **ErrorBoundary**：关键组件有错误边界包裹

---

## 四、数据库修改时必检

- [ ] **迁移幂等性**：迁移逻辑重复执行不会产生副作用
- [ ] **前置条件守卫**：迁移前检查旧数据是否存在，迁移后标记已完成
- [ ] **类型映射**：表结构变更有对应的 TypeScript 接口更新

---

## 五、打包发布前必检

- [ ] **产物干净性**：asar 包中不含 `*.db`、`snapshots/`、`.env`
- [ ] **签名完整性**：逐层签名（.node → .dylib → frameworks → helpers → main app）
- [ ] **entitlements 正确**：包含 `com.apple.security.cs.disable-library-validation`
- [ ] **版本号更新**：`package.json` 中的版本号已更新
- [ ] **新用户体验**：空 userData 目录下应用能正常启动，无残留数据

---

## 六、踩坑后必做（闭环飞轮）

- [ ] **记录经验**：在 `.harness/lessons-learned.md` 中记录本次踩坑
- [ ] **归因分析**：判断根因是缺信息/缺约束/缺验证/缺回退
- [ ] **更新规则**：根据归因更新对应的规则文件或本验证清单
- [ ] **验证闭环**：确认更新后的规则能防止同类错误再次发生
